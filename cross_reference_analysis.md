## 二、矛盾项与裁决

### 矛盾 1：前端框架（最大分歧）

| | A (Perplexity) | B (Deep Research) |
|------|------|------|
| 框架 | Next.js + React | Vue.js |
| UI 库 | shadcn/ui + Tailwind | Element Plus |
| 表单 | React Hook Form + Zod | Element Plus 内置表单 |
| 状态管理 | 未明确 | Pinia |

**分析**：
- A 的选型偏国际化、现代化，shadcn/ui 在移动端体验更好
- B 的选型偏中国企业级应用，Element Plus 中文文档完善，表格/表单组件开箱即用
- B 的 PDF 中给出了完整的组件树、代码示例和部署方案，方案更落地的原因是 Deep Research 做了更深的研究
- 本产品目标用户是**中国市场创业者**，中文文档、表格类组件丰富度、企业级应用成熟度是关键

**裁决**：采用 **Vue.js + Element Plus**（B 方案）
- 理由1：Element Plus 的 Table/Form 组件对财务报表展示是刚需，shadcn/ui 的表格需大量自定义
- 理由2：中文文档完善，降低开发中的沟通和理解成本
- 理由3：B 报告给出了完整到文件的落地细节，可直接进入开发
- 风险提示：如果未来需要更复杂的移动端交互，可以考虑在 Vue 中引入 Naive UI 或 Vant 作为补充

### 矛盾 2：存储方案

| | A (Perplexity) | B (Deep Research) |
|------|------|------|
| 近期 | SQLite + Prisma | IndexedDB（纯本地） |
| 远期 | Postgres | 云同步 |

**分析**：
- A 的 SQLite 方案需要后端，增加部署复杂度
- B 的 IndexedDB 方案零后端，符合隐私合规（中国《个人信息保护法》），且存储量大
- 对于 MVP（单用户、单设备），IndexedDB 足够

**裁决**：采用 **IndexedDB**（B 方案）
- 理由1：零后端部署，MVP 成本更低
- 理由2：用户财务数据存本地，隐私合规风险最低
- 理由3：后期可加导出/导入实现数据迁移，再平滑过渡到云同步

### 矛盾 3：PDF 导出方案

| | A (Perplexity) | B (Deep Research) |
|------|------|------|
| PDF | Puppeteer/Playwright | 页面打印方案 |

**分析**：
- Puppeteer/Playwright 需要 Node 服务端，增加部署复杂度
- 页面打印方案（`window.print()` + CSS `@media print`）纯前端即可
- 对于 MVP，页面打印方案成本最低

**裁决**：采用 **页面打印方案**（B 方案），后期可升级 Puppeteer

### 矛盾 4：开源项目参考

两份报告列出的项目**几乎完全不重叠**，说明两个源的搜索路径不同。

| A 列出 | B 列出 |
|------|------|
| Firefly III (21.3k⭐, PHP+Vue) | Akaunting (8.8k⭐, PHP+Laravel+Vue) |
| Maybe (AGPLv3) | cFlo Cash Flow (Python) |
| Ghostfolio (Angular+NestJS) | GnuCash (C+Scheme) |
| Break Even Calculator (JS) | Planner_Bank (Node.js) |

**共同点**：都推荐了 PHP+Vue 栈的个人财务工具作为主要学习对象（Firefly III ≈ Akaunting）

**裁决**：
- 主要学习 Firefly III（A 推荐，体量最大）的报表结构和 API 设计
- 借鉴 Akaunting（B 推荐）的表单和 UI 交互
- Break Even Calculator 作为盈亏平衡计算的轻量参考实现
- 其余项目参考价值有限或不匹配

### 5.2 计算引擎（确定版）

**统一公式骨架**：
```
利润 = (单位收费 U - 单位变动成本 VC) × 业务量 V - 固定成本 FC

盈亏平衡业务量 V₀ = FC / (U - VC)
目标利润反推：V = (FC + T) / (U - VC)
             U = VC + (FC + T) / V
```

**服务型产能硬约束**：
```
月最大计费小时 = 120h（默认）
月收入天花板 = 时薪 × 120
超过天花板 → UI 预警
```

**三大场景系数**：保守(0.7) / 中性(1.0) / 乐观(1.3)

### 5.3 交互方案（确定版）

```
用户进入
    │
    ├── 选择模式（产品型/服务型）
    │
    ├── 填已知成本（必填：固定成本、变动成本、项目名）
    │
    ├── 主路径 B：输入目标月利润 → 系统反推(P, Q)路径
    │
    ├── 辅路径 A：调整定价/销量滑块 → 实时观察利润变化
    │
    └── 输出：三张报表（P&L、现金流、盈亏平衡图）
         + 服务型额外显示"产能天花板"线
```

### 5.7 已知风险与缓解

| 风险 | 缓解 |
|------|------|
| 计算精度 | decimal.js 处理所有财务运算 + 结果校验 |
