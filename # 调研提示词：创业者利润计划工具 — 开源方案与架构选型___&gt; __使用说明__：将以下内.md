## 一、产品目标

我要搭建一个面向 **初创创业者** 的轻量级 Web 工具。核心理念：

> **用户只管填"花多少钱"，系统负责算"怎么赚回来"**

实际创业者往往只知道成本（房租、工资、材料费等），不知道定什么价、卖多少件才能回本。这个工具的核心价值不是"生成一张静态利润表"，而是**帮用户探索：在给定的成本结构下，什么样的定价策略和销售规模能让项目成立**。

### 支持双创业模式

工具需同时支持两种截然不同的创业形态，用户进入后先选择模式，后续所有计算和展示自动适配：


|  | 模式 A：产品型 | 模式 B：服务型 / 一人公司 (OPC) |
| :-- | :-- | :-- |
| **典型场景** | 实体店、电商、摆摊、批发 | 设计、咨询、开发、个人工作室、自由职业 |
| **销售单位** | 件、杯、单、份 | 小时、项目、客户、月费 |
| **变动成本** | 材料/进货/包装，每件有明确成本 | 近乎为零或极低（获客成本、平台抽成） |
| **固定成本** | 房租、工资、水电、运营费 | 软件订阅、营销、云服务、社保 |
| **启动资金** | 设备、装修、证照、首批库存 | 电脑、软件、注册费、品牌建设 |
| **定价逻辑** | 市场比价 + 成本加成 | 时薪制 / 项目报价 / 月费制 |
| **规模瓶颈** | 受店面/产能/库存限制 | **受个人精力限制**（月计费小时有硬上限） |
| **核心推导变量** | 单价 × 销量 | 时薪/客单价 × 客户数/计费小时 |

**技术上为什么不难**：两种模式的底层计算公式完全一致——只是变量的**名字和单位**不同。用一个计算引擎 + 两套参数配置即可实现，额外开发量约 20%。

**服务型模式的特殊处理**：

- 系统需自动提示"单人产能天花板"（典型月计费小时约 80-120 小时）
- 时薪/客单价的推导更依赖 AI 辅助推荐（服务定价缺乏市场比价基准）
- 支持"项目制"和"月费制"两种收入模型切换

## 二、产品核心架构

### 五大计算引擎（按计算依赖排序）

```
① 启动资金预测 ──→ ④ 销售与成本计划
     │                    │
     └──→ ② 盈亏平衡推导 ←┘
              │
              ▼
          ③ 销售收入预测
              │
              ▼
          ⑤ 现金流量计划
```

| 模块 | 做什么 | 关键计算逻辑 |
| :-- | :-- | :-- |
| **① 启动资金预测** | 用户填项目名+行业+规模，系统估算需要多少启动资金 | 一次性投入（设备/装修/证照）+ 运营备用金（N个月固定成本） |
| **② 盈亏平衡推导（核心）** | 已知成本结构，反推什么单价×什么销量能回本 | 盈亏平衡销量 = 固定成本 / (单价 - 单位变动成本)；生成可探索的 (P, Q) 组合空间 |
| **③ 销售收入预测** | 基于推导出的价格和销量，预测不同场景下的收入 | 保守/中性/乐观三档场景，按月展开 |
| **④ 销售与成本计划** | 按月展开完整的收入-成本-利润计算 | 月收入 - 月变动成本 - 月固定成本 → 月净利润；标注第几个月开始盈利 |
| **⑤ 现金流量计划** | 关注实际现金进出，预警断流风险 | 期初现金 + 现金流入（含回款周期）- 现金流出（含付款周期）→ 期末现金 |

### 三张输出报表

| 报表 | 回答的问题 | 核心字段 |
| :-- | :-- | :-- |
| **利润表 (P\&L)** | 能不能赚钱？ | 营业收入 → 毛利 → 营业利润 → 净利润（含毛利率、净利率） |
| **现金流量表** | 钱会不会断？ | 按月展开：期初现金 + 流入 - 流出 = 期末现金，含预警线 |
| **盈亏平衡分析图** | 卖多少才回本？ | X轴=销量，Y轴=金额，总收入线与总成本线的交叉点 |

### 用户最小输入（≤12 个字段，按模式动态切换标签）

两套模式共用同一套 Schema，但字段标签和含义随模式切换。用 `|` 分隔不同模式下的显示文案。


| 字段 | 产品型文案 | 服务型文案 | 必填 | 用于模块 |
| :-- | :-- | :-- | :-- | :-- |
| 项目名 | 如"写字楼咖啡店" | 如"独立UI设计工作室" | √ | 全部 |
| 行业/品类 | 餐饮/零售/服装 | 设计/开发/咨询 | √ | ①②③ |
| 初始投资估算 | 设备+装修+证照+首批库存 | 电脑+软件+注册费+品牌建设 | 可选 | ①⑤ |
| 月度固定成本 | 租金+工资+水电等 | 软件订阅+营销+社保+云服务 | √ | ②③④⑤ |
| 单位变动成本 | 每件产品的材料/进货成本 | 每客户的获客成本/平台抽成 | √ | ②③④ |
| 定价字段 | 每件产品卖多少钱 | 时薪 / 每客户收费 / 每项目报价 | 可选 | ②③④ |
| 销量字段 | 每月能卖多少件 | 每月计费小时数 / 客户数 | 可选 | ②③④ |
| 目标月利润 | 想赚多少 | 想赚多少 | 可选 | ②③ |
| 收入到账周期 | 现结/月结/N天 | 项目里程碑付款/预付款/尾款 | 可选 | ⑤ |
| 供应商/平台账期 | 供应商结账周期 | 平台结算周期（如有） | 可选 | ⑤ |
| 团队人数/产能 | 员工数 | 每月可投入计费小时数（默认160h） | 可选 | ①② |

### 维度 3：计算模型与数据结构设计（最核心）

**目标**：明确全部计算逻辑的完整数学模型。

请根据标准财务管理逻辑，给出：

#### 3.1 双模式统一公式链

核心洞察：两种模式的数学骨架完全一致，仅变量含义不同。

**通用基础公式**：

```
利润 = (单位收费 U - 单位变动成本 VC) × 业务量 V - 固定成本 FC


产品型：单位收费 = 单价 P，         业务量 = 销量 Q，     VC = 材料成本
服务型：单位收费 = 时薪/客单价 R，   业务量 = 计费小时/客户数 H， VC = 获客成本
```

**统一盈亏平衡推导**：

```
盈亏平衡业务量 V₀ = FC / (U - VC)
盈亏平衡收入 R₀ = FC / (1 - VC/U)


给定目标月利润 T：
  → 所需业务量 V = (FC + T) / (U - VC)
  → 所需单位收费 U = VC + (FC + T) / V


统一约束：U > VC、V > 0
```

**服务型特有约束 — 产能天花板**：

```
月最大计费小时 ≈ 总工作小时 × 计费比例
               ≈ 160h × 50%-75%（含商务、管理、学习等非计费时间）
               ≈ 80-120 可计费小时/月


月收入天花板 = 时薪 × 最大计费小时
             = 时薪 × 120（上限）


系统需在用户设定目标超过天花板时给出预警：
"按你当前的时薪假设，即使满负荷工作（120计费小时），月收入上限为 ¥XX,XXX，请调整时薪或接受现实约束"
```

**服务型收入模型子模式**：

```
时薪制：月收入 = 时薪 × 计费小时
项目制：月收入 = 月均项目数 × 项目均价
月费制：月收入 = 客户数 × 月费 × 续费率
```

**① 启动资金预测公式（按模式）**

```
产品型：
  启动资金 = 设备采购 + 装修费用 + 证照办理 + 加盟费 + 首批库存 + 运营备用金
  运营备用金 = 月度固定成本 × 建议预留月数（通常 3-6 个月）


服务型：
  启动资金 = 电脑设备 + 软件授权 + 公司注册 + 品牌建设(网站/logo) + 运营备用金
  运营备用金 = 月度固定成本 × 建议预留月数（通常 3-6 个月，自由职业建议 6 个月）
```

**② 盈亏平衡推导公式（统一核心）**

```
同上述通用公式，为五种计算模式的核心
```

**③ 销售收入预测公式**

```
统一：月收入 = U × V
多场景：保守(系数0.7) / 中性(1.0) / 乐观(1.3)
服务型附加：标记"产能天花板"线在图表上
```

**④ 销售与成本计划（P\&L）公式**

```
统一公式，同前述
```

**⑤ 现金流量计划公式**

```
统一公式，服务型额外考虑：
- 项目制收款节奏（预付款-里程碑-尾款）
- 平台结算周期（如Upwork/Toptal的周结/月结）
```


#### 3.2 数据模型设计

- 推荐的数据模型 Schema（JSON 或 数据库表结构）
- 需要包含：Project（项目，含 `mode: 'product' | 'service'` 字段）、CostItem（成本项）、Scenario（场景/方案）、MonthlyProjection（月度预测）
- 服务型 Project 需额外字段：`serviceRevenueModel`（hourly/project/retainer）、`maxBillableHours`（产能上限，默认 120）
- 如何处理 "按月" vs "按年" vs "按项目生命周期" 三种计算模式


#### 3.3 反向推导的交互逻辑

核心设计问题：当用户只填了成本，不知道单价和销量时，系统如何引导用户探索可行方案？请给出交互流程设计建议，例如：

- 方案 A：默认展示"盈亏平衡曲面"（不同P×不同Q组合的利润矩阵），用户通过滑块探索
- 方案 B：用户先设目标利润，系统给出多条(P, Q)路径
- 方案 C：结合行业参考数据，给出推荐区间
- 推荐采用哪种组合方案，并说明理由

**特别关注服务型模式的交互差异**：

- 服务型用户对"每客户收费"或"时薪"的感知远弱于产品型用户对"单价"的感知
- 服务型推导更依赖 AI 辅助——根据用户填写的技能/行业推荐定价区间
- 产能天花板（~120计费小时/月）需作为硬约束在 UI 上可视化展示
- 建议默认展示"不同时薪下，需要多少客户/多少计费小时才能回本"的对比表
- 支持用户在"时薪制 / 项目制 / 月费制"之间快速切换，实时看到收入变化

下面给你一版可直接用于立项的研究结论，按你要求的九段结构组织。整体判断是：**MVP 最适合用“Next.js/React + TypeScript + 本地优先计算 + 轻量后端/数据库”路线，核心差异化应放在反向推导交互与服务型定价建模上，而不是重做传统商业计划书软件** 。[^1][^2][^3]

## 1. 执行摘要

最强推荐路径是：前端用 Next.js + React + TypeScript，图表用 ECharts 或 Plotly，表单用 React Hook Form + Zod，计算引擎全部先放在前端/共享包里，后端只负责登录、保存和导出 。[^2][^4]
开源复用上，最值得借鉴的不是“利润计算器”本身，而是 Firefly III、Maybe/Sure、Ghostfolio 这类成熟个人财务产品的架构：它们证明了自托管、报表、图表、规则化数据模型和 API 分层是可行的 。[^3][^5][^4][^2]
就商业成品形态看，LivePlan、Enloop、ProjectionHub 已经验证“创业财务预测”是有付费市场的，但它们更偏静态商业计划和财务预测，你的机会在于做成“输入成本，反推可行售价与销量”的交互式决策工具 。[^6][^7][^8]
对服务型模式，必须把“产能上限”做成硬约束，并把时薪、项目制、月费制切换成第一公民，这会显著提升和产品型工具的差异化 。[^9]
如果时间只有两周，建议先做单人可落地的 MVP：本地存储 + 反向推导面板 + 三张报表 + PDF/Excel 导出 + 中英文 i18n，暂不做多用户协作和复杂 AI 。[^9]

## 2. 现成开源方案对比

目前没有完全命中“创业利润计划 + 反向推导 + 盈亏平衡探索”的大型开源项目，但可以复用个人财务管理、商业工具和 break-even 计算库的能力 。[^10][^1][^2][^3]
下面是更接近目标的一组项目，按“可借鉴程度”排序。


| 项目 | Stars | 技术栈 | 核心功能 | 商用许可 | 二次开发评估 |
| :-- | --: | :-- | :-- | :-- | :-- |
| Firefly III [^2] | 21.3k | PHP, Vue 2, JS | 个人记账、预算、报表、API、自托管 | AGPLv3 [^2] | 适合学数据模型、报表、API 设计；直接复用要注意 AGPL 传染性 |
| Maybe [^3] | 未稳定公开星数，仓库已存在 | 自托管财务应用 | 个人财务、资产跟踪、Docker 部署 [^3] | 仓库说明为 AGPLv3 生态 [^5] | 适合学产品化体验和自托管形态，但要留意项目维护状态 |
| Sure（Maybe 社区 fork） [^5] | 未知 | Maybe 衍生 | 自托管个人财务 | AGPLv3 [^5] | 适合作为参考而非直接依赖，主要看 fork 结构 |

从成熟度看，**Firefly III > Ghostfolio > Maybe/Sure > 轻量 break-even 库** 是更合理的学习路径，因为前两者验证了自托管、报表、图表和数据模型 。[^4][^2]
如果你要一个“可直接落地、商用风险低”的路线，最好不要直接依赖 AGPL 代码做主系统，而是只借鉴其数据结构和交互逻辑，再自己实现核心引擎 。[^5][^2]

## 3. 推荐技术架构

最推荐的 MVP 技术栈是 Next.js + React + TypeScript + Tailwind CSS + shadcn/ui，原因是它兼顾移动端友好、表单组合效率和部署简便 。[^2][^4]
计算引擎建议做成纯 TypeScript 共享包，前端可直接算，后端可复用同一套函数，避免双份逻辑和同步成本 。[^9]
图表建议优先 ECharts：交互强、中文生态成熟、适合折线/柱状/散点/热力图；如果你更重导出和高级可视化，可用 Plotly 作为补充 。[^9]
报表导出建议走 `ExcelJS` 或 `SheetJS` 生成 Excel，PDF 走 `Puppeteer` 或 `Playwright` 打印页面；这比引入复杂报表引擎更适合两周 MVP 。[^9]
存储层如果追求快，先用 SQLite + Prisma + 本地账户；如果更轻，可以先 LocalStorage/IndexedDB，后续再切换到 Postgres 。[^9]

架构上可以理解为四层：

- **交互层**：模式选择、动态表单、参数滑块、结果面板。
- **计算层**：启动资金、盈亏平衡、收入预测、P\&L、现金流。
- **持久化层**：项目、成本项、场景、月度预测。
- **输出层**：图表、表格、PDF、Excel、分享链接。

如果要低代码快速验证，Airtable/NocoDB + 前端模板可以先做需求验证，但你这个产品的核心竞争力在“推导逻辑与交互”，低代码不适合作为最终主架构 。[^9]

## 4. 计算模型与反推设计

你这个产品最关键的数学骨架，就是把产品型和服务型统一成同一个公式：利润 = $(U - VC) \times V - FC$，其中 U 是单位收费，VC 是单位变动成本，V 是业务量，FC 是固定成本 。[^9]
盈亏平衡点就是 $V_0 = FC / (U - VC)$，而目标利润反推就是 $V = (FC + T) / (U - VC)$ 或 $U = VC + (FC + T)/V$ 。[^9]
服务型必须额外加一条硬约束：月最大可计费小时，例如 80–120 小时，并在 UI 上明确告诉用户“不是所有目标都可由努力补足” 。[^9]
收入模型上，服务型要拆成时薪制、项目制和月费制三条分支，否则交互会很难让用户建立感知 。[^9]

更实用的交互组合是：**B + C 为主，A 为辅**。
也就是：先让用户设目标利润，系统给出多条可行路径；再叠加行业参考区间，最后用滑块探索利润曲面和产能约束 。[^9]
这样做比单纯展示盈亏平衡图更适合创业者，因为多数用户并不知道自己应该先问“卖多少”，还是先问“定多少价” 。[^9]

对于服务型模式，默认页面建议展示三块：

- “按当前时薪，要多少小时/多少客户才能回本”。
- “按目标利润，推荐时薪区间”。
- “满负荷 120 小时的收入上限提示”。
这样能把抽象的定价问题转成可感知的产能问题 。[^9]


## 5. 数据模型设计

推荐最小数据模型是 `Project + CostItem + Scenario + MonthlyProjection`，这样足以覆盖模式切换、成本结构、场景模拟和月度现金流 。[^9]
`Project` 至少包含 `mode: 'product' | 'service'`，服务型再加 `serviceRevenueModel` 和 `maxBillableHours` [^9]。
`CostItem` 负责固定成本、变动成本和一次性投入的归类；`Scenario` 负责保守/中性/乐观以及不同价格和销量路径；`MonthlyProjection` 存最终按月展开的数据 。[^9]
按月、按年、按生命周期三种模式最好不要拆成三套引擎，而是统一成一个时间轴，再用聚合粒度控制输出 。[^9]

一个适合 MVP 的 JSON 结构可以是：

```json
{
  "project": {
    "id": "p1",
    "name": "UI design studio",
    "mode": "service",
    "serviceRevenueModel": "hourly",
    "maxBillableHours": 120
  },
  "costItems": [],
  "scenarios": [],
  "monthlyProjections": []
}
```

如果未来要上数据库，Prisma + Postgres 是最稳妥的扩展路线；如果短期只做单机/单用户，SQLite 就够用 。[^9]

## 6. 第三方服务

商业化支付上，Stripe 适合国际化订阅，若偏中国市场则建议优先对接国内支付能力；但具体选型要看你首发市场和收款主体 。[^9]
多语言方面，`next-intl` 或 `i18next` 都适合 React/Next.js 体系，前者更贴近现代 App Router 方案，后者生态更广 。[^9]
导出报表方面，Excel 走 `SheetJS` / `ExcelJS`，PDF 走页面打印方案最简单可靠 。[^9]
如果要接 AI，最值得做的是“项目名/行业 → 参考价格区间/成本项建议”，而不是一开始就做复杂的会计自动识别 。[^9]

关于行业基准数据，这类 API 并没有一个统一的权威开源标准源；更现实的方案是“公开数据集 + 互联网上的价格区间 + 人工整理种子库”混合使用 。[^9]
服务型岗位的时薪基准更适合做成“参考区间库”，来源可以是自由职业平台、招聘市场和社区讨论，但要注意数据质量和版权边界 。[^9]

## 7. MVP 开发路径

Phase 1 建议只做核心闭环：模式切换、动态表单、成本输入、盈亏平衡反推、服务型产能提示、基础图表，这一阶段约 4–6 天 。[^9]
Phase 2 做三张报表、月度现金流、Excel/PDF 导出和本地保存，再用 3–4 天完成 。[^9]
Phase 3 才加入 i18n、登录、云同步、AI 参考区间，剩余时间用来打磨移动端体验和文案 。[^9]
按两周单人开发的节奏，这是比较现实的拆法。

## 8. 商业化建议

最适合的商业模式是 Freemium：免费版支持单项目、基础反推和基础图表，付费版开放多项目、导出、云存储和高级场景 。[^7][^8][^6]
LivePlan、Enloop 和 ProjectionHub 都证明了市场愿意为“财务预测 + 商业计划 + 可视化”付费，而你的差异化是“反向探索”和“服务型定价” 。[^8][^6][^7]
如果要面向中国创业者，最强卖点不是“像财务软件”，而是“帮你从成本倒推出可行收入模型” 。[^9]
定价可以先做低门槛订阅，等验证需求后再卖行业模板、服务型定价包和 AI 推荐额度。

## 9. 风险与注意事项

最大风险是：你做成了一个“更好看的商业计划模板”，却没有真正解决反向推导问题 。[^9]
第二个风险是许可证，Firefly III 和 Maybe/Sure 这类项目带有 AGPL 属性，不能直接照搬到闭源商用主系统里 。[^5][^2]
第三个风险是行业基准数据质量不稳定，尤其服务定价很容易受到地域、平台和个人品牌影响，AI 只能做参考，不宜做强结论 。[^9]
最后，服务型模式的产能约束一定要显式化，否则用户会误以为收入可以无限线性增长 。[^9]

如果你下一步要推进，我建议直接进入“开源项目清单 + 技术栈落地方案 + 计算模型 JSON Schema”三件事的详细版。
