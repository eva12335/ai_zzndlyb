## 一、产品定位

### 1.1 一句话描述

帮 AI 时代的超级个体从"我适合一人公司吗？"到"我知道每个月要花多少钱"出发，反向推导定价、销量和盈亏平衡的交互式利润探索工具。

**核心文案**：
> 你适合一人公司吗？心里没谱
> 成本门儿清，回本没底？
> 丢个数，我们帮你算明白

### 1.2 目标用户

| 用户类型 | 典型场景 | 痛点 |
|---------|---------|------|
| **产品型用户** | 开实体店、做电商、摆摊 | 知道房租和进货价，不知道卖多少钱、卖多少件才回本 |
| **服务型/一人公司** | 自由设计师、独立开发者、咨询顾问 | 知道每月固定开销，不知道怎么定时薪、需要几个客户才收支平衡 |
| **AI 时代超级个体** | Vibe Coder、自媒体博主、AI 创作者 | Token 成本、工具订阅是新的固定支出，传统财务模板不覆盖 |

### 1.3 核心差异化

传统工具（LivePlan、Enloop）是"正向填表 → 生成利润表"，用户需要提前知道收入数据。

一人公司罗盘是"**填入成本 → 反向探索可行方案**"：定价和销量可以留空，系统帮你推导。核心输入字段中定价和销量均标注"可选"。

## 二、用户故事

### 2.1 产品型用户

```

他知道：
  - 月租金+工资+水电 ≈ ¥20,000

  - 每月 AI 客服工具订阅 ≈ ¥500
他不知道：
  - 每杯该卖多少钱？
  - 一个月能卖多少杯？
  - 能不能回本？

小明打开小程序 → 默认产品型 → 填入已知成本 → 系统展示：
  "如果定价 ¥25/杯，每月需卖 1,026 杯才能回本"

  "如果你想月赚 ¥10,000，定价 ¥30 的话需要卖 1,333 杯/月"
```

## 四、计算模型

### 4.1 统一公式

```
利润 = (U - VC) × V - FC - AC

U  = 单位收费      产品型=单价        服务型=时薪
VC = 单位变动成本   产品型=每件材料+获客  服务型=0（服务型无单位成本）
V  = 业务量       产品型=月销量       服务型=月计费小时
FC = 月度固定成本 + Token（月）
AC = 获客支出       产品型=0（已含在VC中）  服务型=获客成本/客户 × 月拉新客户数
P  = 目标月利润

预计月总花费 = FC + AC
```

| | 产品型 | 服务型 |
|--|--------|--------|
| 获客成本归属 | 并入"每件材料成本"，用户自行合计 | 独立字段"获客成本/客户"，按"月拉新客户数"折算为月支出 |
| AC 计算 | 0（已在 VC 中） | 获客成本/客户 × 月拉新客户数（默认 1） |
| 为什么不同 | 产品型获客按件摊销（每卖一杯含 ¥X 推广费） | 服务型获客按客户计（每拉一个客户花 ¥X） |

### 4.2 核心推导

| 计算目标 | 公式 | 说明 |
|---------|------|------|
| 盈亏平衡业务量 | V₀ = FC / (U - VC) | 利润为零时的业务量 |
| 盈亏平衡收入 | R₀ = U × V₀ = FC / (1 - VC/U) | 利润为零时的收入 |
| 目标利润所需业务量 | V = (FC + P) / (U - VC) | P = 目标月利润 |
| 目标利润所需定价 | U = VC + (FC + P) / V | P = 目标月利润 |

> 符号约定：U = 单位收费，VC = 单位变动成本，V = 业务量，FC = 固定成本，**P = 目标利润**，R = 收入

### 4.3 服务型产能约束

```
月最大计费小时 = 120h（默认，可调整）
月收入天花板 = 时薪 × 120
超过天花板 → UI 橙色预警
```

### 4.4 三场景系数

| 场景 | 收入系数 | 用途 |
|------|---------|------|
| 保守 | 0.7 | 最坏情况 |
| 中性 | 1.0 | 基准预测 |
| 乐观 | 1.3 | 最好情况 |

### 5.4 🆕 输入校验（四层防御）

**设计原则**：垃圾数据在进入计算引擎前被拦截，不等到提交才报错。事前预防 > 事后补救。

**第一层：输入控件（前端原生）**

- `<input type="number" min="0" max="99999999">` — 阻止负数和非数字
- 小程序数字键盘 — 阻止非数字字符
- `max` 限制 — 阻止天文数字（¥99,999,999 为上限）

**第二层：失焦校验（即时反馈，`onBlur`）**

| 校验规则 | 触发条件 | 反馈方式 |
|---------|---------|---------|
| 必填为空 | `*` 字段失焦时值为空 | label 标红 + 输入框红框 + 下方红色提示"请填写" |
| 必填为零 | `*` 字段值为 0 | label 标红 + "固定成本不能为 0" |
| 值过大 | 任意金额 > ¥10,000,000 | 黄色警告"金额较大，请确认"，**不阻断** |
| 变动成本 ≥ 定价 | VC > 0, U > 0, U ≤ VC | 橙色警告"定价 ≤ 成本，每卖一件亏 ¥X"，**不阻断** |

**第三层：计算前校验（`validateInput.ts`，纯函数）**

```typescript
interface ValidationResult {
  pass: boolean;           // true=可计算，false=阻断
  level: 'ok' | 'warn';   // warn=可计算但有提醒
  errors: ValidationError[];  // 阻断性错误
  warnings: ValidationError[]; // 非阻断警告
}

function validateInput(fields: CostFields): ValidationResult {
  const errors: ValidationError[] = [];
  const warnings: ValidationError[] = [];

  // 阻断规则
  if (!fields.fixedCost || fields.fixedCost <= 0)
    errors.push({ field: 'fixedCost', msg: '固定成本必须大于 0' });
  if (fields.mode === 'product' && (!fields.unitVariableCost || fields.unitVariableCost <= 0))
    errors.push({ field: 'unitVariableCost', msg: '每件材料成本必须大于 0' });
  if (fields.fixedCost > 10_000_000 || fields.unitVariableCost > 10_000_000)
    errors.push({ field: 'fixedCost', msg: '金额超过上限' });

  // 警告规则（不阻断）
  if (fields.unitPrice > 0 && fields.unitVariableCost > 0 && fields.unitPrice <= fields.unitVariableCost)
    warnings.push({ field: 'unitPrice', msg: `定价 ≤ 成本，每卖一件亏 ¥${fields.unitVariableCost - fields.unitPrice}` });
  if (fields.tokenCost > fields.fixedCost * 2)
    warnings.push({ field: 'tokenCost', msg: 'Token 费用超过固定成本 2 倍，请确认' });

  return {
    pass: errors.length === 0,
    level: warnings.length > 0 ? 'warn' : 'ok',
    errors,
    warnings,
  };
}
```

**第四层：计算中保护（`breakEven.ts`）**

```typescript
// 除零保护
if (contributionMargin.eq(0)) {
  return { error: '单位贡献毛利为零——定价不能等于变动成本' };
}

// 无穷大检查
const bev = totalFC.div(contributionMargin);
if (bev.gt(999_999)) {
  return { error: '盈亏平衡量异常，请检查输入', value: bev };
}
```

**校验时机**：

```
用户填完一个字段 → onBlur → 第二层失焦校验（即时反馈）
    ↓
用户修改滑块/切换模式 → 第三层 validateInput() → 
    ├── pass=true  → calculateAll() 正常执行
    ├── pass=true + warnings → calculateAll() 执行 + 顶部黄条显示警告
    └── pass=false → 不执行计算，错误字段标红，底部显示"请修正后再试"
```

### 6.1 利润表（P&L）🆕 含税种切换

| 项目 | 计算来源 |
|------|---------|
| 一、营业收入 | P × Q |
| 减：营业成本 | VC × Q |
| 二、毛利 | 收入 - 成本 |
| 毛利率 | 毛利 / 收入 × 100% |
| 减：固定费用 | 月度固定成本 |
| 减：获客支出 | 产品型=0（已在VC中）；服务型=获客成本/客户 × 月拉新数 |
| 减：Token | 月 AI 工具费 |
| 减：折旧摊销 | 初始投资 / 分摊月数 |
| 三、营业利润 | 毛利 - 固定费用 - 折旧 |
| 🆕 减：增值税 | 营业收入 × 增值税率（小规模纳税人默认 1%） |
| 🆕 减：所得税 | 根据税种切换自动计算 |
| 四、净利润 | 营业利润 - 增值税 - 所得税 |
| 净利率 | 净利润 / 收入 × 100% |

**🆕 税种切换**（用户在成本输入页预设）：

| 主体类型 | 增值税 | 所得税计算 | 典型用户 |
|---------|--------|----------|---------|
| 个体工商户 | 小规模 1%（季 30 万以下免） | 经营所得 5%-35% 累进 | 实体店、摆摊 |
| 有限公司 | 小规模 1% | 企业所得税 5% + 分红 20% | 有融资/招人计划 |
| 个人（默认） | 无 | 劳务报酬 20%-40% | 自由设计师、Coder |

> 用户只需选一次主体类型，系统自动匹配税率。利润表底部显示"如果你注册个体户，每年多留 ¥X,XXX"等对比提示。

### 6.5 🆕 利润诊断建议（规则引擎 · 纯本地）

**设计原则**：利润为负时不调大模型，由规则引擎在本地做根因诊断。三类根因覆盖所有亏损场景，建议文案预写、零幻觉、零延迟。

**触发条件**：净利润 < 0（ROI 页实时计算后自动触发）

**根因诊断树**（三分类）：

```
净利润 < 0
  │
  ├── 毛利 ≤ 0（定价 ≤ 变动成本）
  │     → 根因：单位亏损。每卖一件/每工作一小时都在贴钱。
  │     → 建议方向：提价（产品型）/ 提时薪（服务型），降变动成本
  │
  ├── 毛利 > 0 但 毛利 < 固定成本 + Token
  │     → 根因：规模不足。毛利不够覆盖固定开支。
  │     → 建议方向：增量（提销量/拉客户），或降固定成本
  │
  └── 毛利 > 固定成本 + Token 但净利润 < 0
        → 根因：额外负担。获客成本或税费侵蚀利润。
        → 建议方向：优化获客渠道降低 AC，或切换税种
```

**诊断输出 Schema**：

```typescript
interface ProfitDiagnostic {
  rootCause: 'unitLoss' | 'scaleGap' | 'extraBurden';
  rootCauseLabel: string;          // "单位亏损" / "规模不足" / "额外负担"
  diagnosis: string;               // 1-2 句诊断说明，引用用户的实际数字
  suggestions: SuggestionItem[];   // 2-3 条具体建议，按优先级排序
}

interface SuggestionItem {
  action: string;                  // 具体行动（如"将定价从 ¥18 提至 ¥25"）
  expectedImpact: string;          // 预期效果（如"预计扭亏为盈，月净利 ¥850"）
  difficulty: 'low' | 'medium';   // 执行难度
  category: 'price' | 'cost' | 'volume' | 'tax' | 'acquisition';
}
```

**建议生成规则**（纯函数，所有文案预写）：

| 根因 | 建议优先级 | 示例文案 |
|------|----------|---------|
| unitLoss | 1.提价 2.降变动成本 3.停售 | "你的定价 ¥18 低于变动成本 ¥22，每卖一件亏 ¥4。建议提价至 ≥ ¥25。" |
| scaleGap | 1.增量 2.降固定成本 3.提价辅助 | "毛利 ¥3,000 不够覆盖固定成本 ¥5,500。需月销增至 110 件（+38%），或降固定成本 ¥2,500/月。" |
| extraBurden | 1.降获客成本 2.切换税种 3.提价补税 | "获客支出 ¥1,200/月吃掉大部分利润。建议尝试自然流量渠道（如内容营销），将 AC 控制在 ¥300/月以内。" |

**结合测评结果**（有测评数据时增强）：

- 测评得分中"风险承受"≥6 且根因为 unitLoss → 追加："你的风险承受力较强，可考虑激进提价 30%-50% 测试市场反应。"
- 测评得分中"谈判技巧"<5 且根因为 unitLoss → 追加："谈判力是你的弱项，提价时建议先练习'价值报价'话术。"
- 测评得分中"市场和客户关系"<5 且根因为 scaleGap → 追加："获客是你的短板，建议从即刻/小红书等内容渠道做自然引流。"

## 八、智能模块

### 8.1 智能定价推荐 【V2 延后】

**触发时机**：成本输入页面，用户已填写 ≥3 个必填字段但定价/销量留空时，"智能推荐"按钮亮起。

**交互流程**：

```
用户填写成本字段 → 点"🤖 智能推荐"
    → L1 输入校验（拒绝垃圾数据：负数/零/极端值）
    → L2 云函数调用 AI → 返回推荐结果
    → L3 规则验证（数学交叉校验：毛利>0、毛利率0-1、盈亏平衡>0）
    → L4 不通过则降级为本地推荐
    → L5 结果卡片弹出（底部滑入）
        ├── 📊 推荐定价区间（金色高亮）
        ├── 🧮 推导过程：成本 × 倍率 → 价格、毛利、毛利率
        ├── 📦 回本所需业务量
        ├── 💡 AI 理由（2-3 句）
        └── [应用推荐] [换一个] [关闭]
    → 用户点"应用推荐" → 定价和销量自动填入 → 滑块跳至推荐位置
```

**五层可信度控制**（详见 TECH_DESIGN 3.5.1）：

| 层 | 机制 | 说明 |
|---|------|------|
| L1 输入校验 | 拒绝负数/零/极端值 | 没填成本不让调 AI |
| L2 AI 生成 | 云函数调用大模型 | DeepSeek-v3.2 默认 |
| L3 规则验证 | 数学交叉校验 | profitPerUnit>0、毛利率 0-1、盈亏平衡>0 |
| L4 本地降级 | 可配置倍率兜底 | 倍率可调（默认3），非硬编码行业经验 |
| L5 用户调整 | 换一个 + 手动改 | 最终决策权在用户 |

**云函数输入 Schema**：

```typescript
interface PricingRecommendationInput {
  mode: 'product' | 'service';
  fixedCost: number;                    // 月度固定成本（不含获客）
  unitVariableCost?: number;            // 单位变动成本（产品型）
  tokenCost: number;                    // AI 工具月费
  acquisitionCost?: number;             // 获客支出/月
  targetProfit?: number;                // 目标月利润（可选）
  industry?: string;                    // 行业提示（可选，如"写字楼咖啡"）
}
```

**职责拆分**：AI 判断倍率 + 写人话，系统用倍率算所有数字。

| 谁算 | 字段 | 理由 |
|------|------|------|
| **AI** | `multiplier`、`reasoning`、`riskNote` | 倍率需要行业常识判断，人话和风险提示必须 AI |
| **系统** | `recommendedPriceLow/High`、`grossProfitPerUnit`、`grossMargin`、`breakEvenVolume` | 纯数学，`breakEven.ts` 公式比 AI 更准 |

**云函数输出 Schema**（AI 只返回这三个字段）：

```typescript
interface AiPricingOutput {
  multiplier: number;         // AI 判定的倍率（2-8），前端用它算价格
  reasoning: string;          // 推荐理由（1-2 句，必须引用输入数据）
  riskNote?: string;          // 风险提示（只能引用输入数据中的字段）
}
```

**系统计算**（`useAiRecommend.ts` 拿到 AI 输出后执行）：

```typescript
const priceLow  = unitVariableCost × multiplier × 0.9;  // 下限
const priceHigh = unitVariableCost × multiplier × 1.1;  // 上限
const grossProfitLow = priceLow - unitVariableCost;
const marginLow = grossProfitLow / priceLow;
const bevLow = (fixedCost + tokenCost) / (priceLow - unitVariableCost);
// ... high 同理
```

**L2 System Prompt 约束**：

```
你是一人公司定价顾问。你只需要判断一个数字：倍率。
其余所有金额由系统计算，你不用管。

必须遵守：
  - multiplier 必须在 2 到 8 之间（产品型）
  - multiplier 必须在 1.5 到 5 之间（服务型）
  - multiplier 的推导必须基于 costBase，在 reasoning 中说明
  - reasoning 不超过 2 句，必须引用输入数据中的具体数字

riskNote 只能引用用户输入中的成本、销量、增长率、固定成本等字段。
禁止引用市场、行业、消费者、竞争对手等用户未提供的信息。

禁止：
  - 编造市场数据、行业对标、竞品价格
  - 使用"研究表明""行业平均""通常来说"
  - 推荐具体商业策略（如"你应该做外卖"）

温度：0.3，最大 tokens：300
严格按 AiPricingOutput JSON Schema 返回。
```

**降级策略**：
- AI 超时（默认 8s，可配置 `AI_TIMEOUT_MS`）或 L3 校验失败 → 本地降级引擎
- 产品型：推荐定价 = 变动成本 × 可配置倍率（默认 3，运营可调）
- 服务型：推荐时薪 = 月固定成本 ÷ 可配置工时（默认 80h）
- 降级结果标注"本地估算"，引导用户网络恢复后重新生成

### 8.2 🆕 AI 深度报告（付费版 · 大模型驱动）

**核心差异**：免费版"你的画像"告诉用户**你是什么**——分数、标签、段位。AI 深度报告回答免费版说不透的问题——**你的画像对你具体这门生意意味着什么、能多赚多少钱**。

**触发时机**：画像页点击"生成 AI 深度报告"。任何时候都可触发——成本数据有无皆可，有则解锁财务计算。

**两条分支**：

| 条件 | 报告内容 | 价值 |
|------|---------|------|
| 用户未填成本 | 段位分析 + 机会方向（引导去填 ROI） | AI 告诉你"为什么你是这个段位，最大杠杆在哪" |
| 用户已填成本 | 段位分析 + 机会计算（具体金额）+ 路线图 | AI 算得出"谈判力 3→4，材料成本能降 ¥840/月" |

**报告结构**：

```
🧭 段位分析     ← AI 综合 11 维度 + 业务模式 + 成本数据，
                  给出因果链人格解释——为什么你是谨慎型技术派？
                  这个段位在一人公司路上最常踩什么坑？
                  （规则引擎做不到的综合分析）

💰 机会雷达     ← 11 维度 × 成本数据 × 业务模式：
                  有成本 → 每个短板改 1 分 = 具体金额（如 谈判力 3→4 = 月多 ¥840）
                  无成本 → 方向引导（"[填入成本数据解锁具体金额 →]"）
                  按 ROI 排序

🎯 优先行动     ← Top 3 行动，每条绑金额 + 难度 + 风险标注
                  每条标注来源维度（← 谈判力 3/10）

📈 改进路线图   ← 3/6/12 个月目标 + 预期画像变化 + 财务变化
                  （有成本数据才显示，无成本则隐藏此段）
```

**幻觉防护**（四层 + Prompt 完整约束）：

**第一层：AI 不算钱，只做翻译**

机会雷达和路线图中的金额由本地规则引擎计算（与 ROI 页 `breakEven.ts` 同一公式），AI 只负责将数字翻译成人话。Prompt 中传入的 `costData` 已包含预计算的财务影响值，AI 不能修改或编造。

**第二层：System Prompt 锁定边界**

```
你是一人公司画像分析师（Entrepreneurial Persona Analyst）。

══════════════ 核心原则 ══════════════
1. 所有结论必须直接来源于输入数据。
2. 如果无法从输入数据推导，不要输出该结论。
3. 不允许补充用户未提供的信息。
4. 不允许推断职业、行业、收入、家庭情况、商业模式。
5. 不允许推断人格特质（除非维度直接测量该特质）。
6. 不知道就不说——省略结论比编造结论更好。

══════════════ 分析规则 ══════════════
优势分析：
  仅选择得分最高的 2~3 个维度。
  每个优势结论必须引用至少一个维度得分。
  示例：正确 → "独立动机 9 分，说明你具有较强的自主驱动力。"
        错误 → "你是一个执行力很强的人。"

机会雷达：
  仅选择得分最低的 2~3 个维度。
  每个短板分析必须引用该维度的当前分值。
  有 costData 时附加预计算的 financialImpact，无 costData 时 financialImpact 为空字符串 ""。

综合画像（personaSummary、overallAssessment）：
  必须同时引用至少两个维度得分作为依据。
  禁止仅凭单一维度生成整体画像。
  示例：正确 → "独立动机 9 分 + AI 技能 8 分 → 具备独立开展业务的基础条件"
        错误 → "独立动机 9 分 → 你一定适合创业"

行动建议：
  必须对应低分维度，每条建议标注来源维度名。
  数量：固定 3 条。

══════════════ 禁止行为 ══════════════
- 禁止编造用户未提供的财务数字、市场数据、行业对标
- 禁止推荐具体商业平台、社群名称、课程、工具品牌
- 禁止给出"你应该辞职""你应该贷款"等人生决策建议
- 禁止维度外推——把某个维度得分扩展为未测量的特征
  示例：谈判力低 ≠ 不善沟通、风险承受低 ≠ 害怕失败、决策力低 ≠ 缺乏领导力

══════════════ 格式约束 ══════════════
- 严格按照 `AiReportOutput` JSON Schema 返回，不得新增字段
- 用户无成本数据时，financialImpact 字段必须为空字符串 ""
- 用户无成本数据时，roadmap 字段必须为 null
- 温度：0.3，最大 tokens：1,500
```

```typescript
function validateAiReport(output: AiReportOutput, input: AiReportInput): boolean {
  // 段位名必须匹配输入
  if (output.segmentLabel !== input.segmentLabel) return false;
  // 每个 opportunity 的维度名必须在 11 维列表内
  if (output.opportunities.some(o => !DIMENSION_NAMES.includes(o.dimension))) return false;
  // 有成本数据时，financialImpact 金额与本地计算结果误差不超过 5%
  if (input.costData) { /* 交叉校验 */ }
```

## 九、用户身份与商业化

### 9.1 核心原则：不设登录，不设注册

用户打开小程序就能用，从始至终**见不到"登录""注册"两个字**。

原因：
1. 微信已经知道用户是谁——`wx.login()` 静默获取 openid，无需用户操作
2. 这是个本地计算工具——核心功能不依赖服务端，不需要账号体系
3. 加登录页反而坏事——工具类小程序弹授权框，转化率跌 30%+

### 9.2 静默用户识别

```
小程序启动 (app.tsx)
    │
    ├── 1. 调用 Taro.login() 获取 code（无弹窗，无用户感知）
    ├── 2. 本地生成匿名标识，区分不同用户
    ├── 3. 全程不阻塞 UI，失败不影响功能
    │
    └── 前端永远不出现"登录"相关 UI
```

- 用户看到的是：打开 → 直接用
- 后台做的是：获取 openid，用于未来云端同步和商业化统计
- 个人主体小程序限制：**不做手机号获取**（个人主体不支持）
